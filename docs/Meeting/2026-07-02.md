# Meeting Objective

Officially start the EPSM Manager project.

The objective of this meeting was to establish the long-term vision of the platform, define the project foundations and agree on the architectural principles before starting software development.

---

# Context

È Pronto Si Mangia currently manages production using Excel spreadsheets and handwritten notes.

The business requires a digital solution capable of replacing manual data entry while preserving existing business rules and improving operational traceability.

The existing Excel workbook contains valuable business knowledge that must be preserved during the digital transformation.

---

# Decisions Made

## Project Vision

EPSM Manager will become the operational platform of È Pronto Si Mangia.

The long-term objective is to replace manual operational processes through a cloud-based web platform.

Excel will initially remain as the analytics layer but will gradually be replaced by native web dashboards.

---

## Cloud Architecture

The following architecture was approved:

GitHub Pages

↓

Web Application

↓

Supabase

↓

Analytics Layer

↓

Future Web Dashboard

---

## Technology Stack

Approved technologies:

- GitHub Pages
- HTML
- CSS
- JavaScript
- Supabase
- PostgreSQL
- Excel (Transition Phase)

---

## Development Strategy

Development will follow incremental Sprints.

Every Sprint must deliver business value.

Documentation always precedes implementation.

---

## Primary Business Entity

The Dough Batch becomes the central business entity.

Every production record belongs to one Dough Batch.

Products, operators and remaining dough are associated with each Dough Batch.

---

## Business Principles

The following principles were officially adopted:

- Business First
- KISS
- Security by Design
- Documentation First
- Zero Cost First
- Scalability
- Modular Architecture
- Single Source of Truth
- Progressive Improvement
- Digitalize Processes
- Traceability

---

## Security

Supabase becomes the trusted security boundary.

GitHub Pages only hosts the application.

Operational data will never be stored in GitHub.

Authentication will be managed by Supabase.

---

## User Model

Production will initially use a shared production account.

Business traceability will be achieved by selecting the operators participating in each Dough Batch.

Editing operations will record the operator responsible for the modification.

---

## Data Strategy

Supabase becomes the only operational database.

Business facts are stored.

Business indicators are calculated.

Operational history should always be preserved.

---

## Documentation

The project documentation structure was officially created.

Documentation becomes part of the product.

Every future architectural decision must be documented.

---

# Risks Identified

Current risks include:

- Business rules still evolving.
- Dough inventory process requires further analysis.
- Remaining dough workflow requires validation.
- Bonus calculation requires additional business clarification.
- Future authentication model may evolve.

All identified risks have been accepted and documented.
